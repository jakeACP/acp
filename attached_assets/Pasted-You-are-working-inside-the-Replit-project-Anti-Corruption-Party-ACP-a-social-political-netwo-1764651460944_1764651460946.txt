You are working inside the Replit project:
“Anti-Corruption Party” (ACP) – a social/political network with auth, posts, profiles, groups, etc.

GOAL:
Design and fully implement a robust “friends” system that works end-to-end across the entire site (web + mobile views) without breaking existing features.

GENERAL APPROACH:
1. First, scan the codebase to understand:
   - Tech stack (backend framework, database, frontend framework).
   - Existing User model and any follow / contact / relationship logic.
   - Existing routes/pages for:
     - User profile
     - Timeline / feed
     - Notifications
     - Search / “Find Friends” or “People” features
     - Mobile layout / navigation.
2. Work incrementally and keep all existing functionality working. Re-run tests and dev server often.

==================================
DATA MODEL & BACKEND REQUIREMENTS
==================================

Add a first-class “friends” system with the following states:

- `pending` (outgoing request)
- `requested` (incoming request)
- `accepted` (friends)
- `blocked` (optional but preferred)

Implement using whatever pattern fits the current stack best (e.g., a `Friendship` table, a many-to-many join table, or a friendship collection). It should support:

- `requester_id`
- `receiver_id`
- `status`
- timestamps
- basic uniqueness constraints (no duplicate relationships, no self-friendship).

Add backend APIs / routes (REST or GraphQL, matching existing style):

1. Send friend request
   - Input: target user id or username/email.
   - Auth required.
   - Validate: cannot send to self, cannot duplicate existing friendship or pending request.

2. Accept friend request
   - Only the receiver can accept.
   - Convert `pending/requested` → `accepted`.

3. Decline / cancel friend request
   - Receiver can decline; requester can cancel outgoing request.

4. Unfriend
   - Either side can remove an accepted friendship.

5. Block / unblock (if consistent with app norms)
   - Optional but supported if the rest of the codebase uses blocking.
   - Block should prevent new requests and hide friendship.

6. List friends
   - `/api/friends` → list of accepted friends for current user.
   - Pagination if needed to match existing API patterns.

7. List friend requests
   - Incoming (`status=requested`)
   - Outgoing (`status=pending`)

8. Search for new friends
   - Reuse existing user search if present; if not, add a simple search by:
     - username
     - email
     - first/last name (if fields exist).

Secure all endpoints:
- Require auth middleware.
- Enforce proper ownership checks.
- Avoid leaking email or private info beyond what’s already visible elsewhere in the app.

Add basic unit/integration tests for:
- Send/accept/decline/cancel/unfriend flows.
- Attempted duplicates.
- Permissions (cannot accept someone else’s request, etc.).

3. **Find Friends / Search**
