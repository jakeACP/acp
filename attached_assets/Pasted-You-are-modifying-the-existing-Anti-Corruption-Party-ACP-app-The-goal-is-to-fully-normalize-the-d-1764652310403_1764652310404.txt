You are modifying the existing Anti-Corruption Party (ACP) app. The goal is to fully normalize the database and build a robust, auto-populating Politician Profile + Corruption Scorecard system driven by ZIP code lookup and ChatGPT API enrichment.

Assumptions:
- The project already has users, authentication, and basic app structure.
- We can call the OpenAI ChatGPT API using an environment variable like OPENAI_API_KEY.
- Tech stack: use whatever is already in the project (Node/Express + PostgreSQL, or equivalent). If unsure, inspect the repo and stay consistent.

--------------------------------
1. DATABASE NORMALIZATION
--------------------------------
Refactor / extend the database to be properly normalized around geography, political offices, and politicians.

Create or ensure these core tables (name them clearly and consistently):

1) Geographic + Office Structure
- zip_codes
  - id (PK)
  - zip_code (string, unique)
  - city
  - state
  - county

- political_offices
  - id (PK)
  - name (e.g., "U.S. Senator", "U.S. Representative", "Governor", "State Representative", "Mayor", "City Council Member", etc.)
  - level (federal / state / county / city / local)
  - chamber_or_type (e.g., "House", "Senate", "Executive", "Judicial", etc.)
  - description (text)
  - term_length_years (int, nullable)

- office_geo_mappings
  - id (PK)
  - political_office_id (FK -> political_offices.id)
  - zip_code_id (FK -> zip_codes.id)
  - district_name_or_number (string, nullable)
  - jurisdiction_name (e.g., “MN-03”, “Hennepin County”, “City of Minneapolis”)
  - UNIQUE(political_office_id, zip_code_id, district_name_or_number)

This structure ensures that political positions are FIXED per ZIP code / district and not duplicated on every profile.

2) Politicians & Terms in Office
- politicians
  - id (PK)
  - full_name
  - party
  - official_email (string, nullable, used for claim verification)
  - official_website (nullable)
  - photo_url (nullable)
  - biography (text, nullable – filled by ChatGPT)
  - created_at
  - updated_at

- politician_office_terms
  - id (PK)
  - politician_id (FK -> politicians.id)
  - political_office_id (FK -> political_offices.id)
  - office_geo_mapping_id (FK -> office_geo_mappings.id, nullable if not needed)
  - start_date (nullable)
  - end_date (nullable)
  - is_current (boolean)
  - UNIQUE(politician_id, political_office_id, office_geo_mapping_id, is_current) where only one current term per office/geo mapping

3) Corruption Scorecard Data
- politician_scorecards
  - id (PK)
  - politician_id (FK -> politicians.id)
  - overall_corruption_score (numeric, nullable; can be computed)
  - avg_campaign_donation (numeric, nullable)
  - last_refreshed_at (timestamp – when we last pulled data via ChatGPT/API)

- sponsors
  - id (PK)
  - name
  - industry (nullable)

- politician_sponsors
  - id (PK)
  - politician_id (FK -> politicians.id)
  - sponsor_id (FK -> sponsors.id)
  - total_amount (numeric, nullable)
  - rank (int – 1 through 10 for top sponsors)

- votes
  - id (PK)
  - politician_id (FK -> politicians.id)
  - bill_name
  - bill_identifier
  - date
  - position (e.g., "Yea", "Nay", "Present", "Absent")
  - summary (text, nullable)
  - category (e.g., “Healthcare”, “Defense”, etc.)

4) User-Generated Corruption Ratings & Article Reviews
- users (already exists; DO NOT duplicate – just reference it)

- news_articles
  - id (PK)
  - politician_id (FK -> politicians.id)
  - title
  - url
  - source_name
  - published_at (timestamp)
  - summary (text, nullable – generated by ChatGPT)
  - is_recent (boolean; used for “top 3 recent” display)

- corruption_votes
  - id (PK)
  - politician_id (FK -> politicians.id)
  - user_id (FK -> users.id)
  - news_article_id (FK -> news_articles.id, nullable)
  - vote_value (int: +1 for “Corrupt”, -1 for “Not Corrupt” OR inverse; document convention)
  - comment (nullable)
  - created_at
  - UNIQUE(user_id, politician_id, news_article_id) to prevent spam voting on same article/politician combo

5) Profile Claiming
- politician_claims
  - id (PK)
  - politician_id (FK -> politicians.id)
  - user_id (FK -> users.id)
  - official_email (snapshot of email used)
  - verification_token
  - status (enum: "pending", "approved", "rejected")
  - created_at
  - verified_at (nullable)

Ensure all tables have foreign keys, indexes on frequently queried fields (zip_code, politician_id, political_office_id), and no duplicated denormalized blobs of the same data.

2. ZIP-BASED LOOKUP & AUTO-CREATION FLOW
--------------------------------
Implement a flow for when a user searches their ZIP code:

1) User enters ZIP code on the “Find Your Reps” page.
2) Backend:
   - Look up or insert the ZIP in zip_codes.
   - For that zip_code_id, query office_geo_mappings and political_offices to get the list of offices relevant to that zip.
   - For each office_geo_mapping:
     - Check if there is a current politician_office_term with is_current = true.
     - If NO politician exists yet for that office/geo:
       - Trigger a background function that calls the ChatGPT API (and any other civic data APIs if available) to discover:
         - The current office holder’s full name, party, official email, official website, photo URL, and basic biography.
       - Create a politicians row and a politician_office_terms row.
       - Also stub a politician_scorecards row with null fields.
   - Return the list of offices and linked politicians (once created) to the frontend.

Design this so that missing politicians are created ON DEMAND when people search, gradually building the politicians database over time.

3. POLITICIAN PROFILE + CORRUPTION SCORECARD PAGE
--------------------------------
Build a dedicated Politician Profile page component (e.g., route: /politician/:id).

When this page is opened for a given politician_id:

1) Check the politician_scorecards.last_refreshed_at.
   - If null or older than a configured threshold (e.g., 7 days):
     - Show a loading state / skeleton UI while data is fetched.
     - Call a backend endpoint like: POST /api/politicians/:id/refresh-scorecard

2) On backend scorecard refresh:
   - Use ChatGPT API to:
     - Summarize the politician’s corruption-related reputation.
     - Identify top 10 sponsors (with sponsor names and industries).
     - Estimate/derive average campaign donation.
     - Summarize recent voting record with key bills and positions.
     - Provide 3–10 recent news article URLs + metadata:
       - title, source, publish date, brief summary, classification of whether article suggests corruption or not.
   - Insert/update:
     - sponsors and politician_sponsors (top 10 sponsors).
     - votes records for recent key votes.
     - news_articles (mark a subset “is_recent = true” for top 3).
     - politician_scorecards:
       - overall_corruption_score (e.g., 0–100, where higher = more corrupt, or vice versa).
       - avg_campaign_donation.
       - last_refreshed_at (now).

   IMPORTANT: The scorecard should only be refreshed when necessary, not on every page load, to reduce API usage.

3) On frontend Politician Profile page, display:

   Header:
   - Politician photo, name, party, office title, district/jurisdiction.
   - Current overall corruption score (big visual element, gauge/bar).

   Claim Section:
   - “Is this your official profile?” CTA button.
   - Clicking opens a dialog to enter an official email address (must match or be reasonably close to politician.official_email if known).
   - After entering email, hit backend endpoint to start claim process (see Claiming section below).

   Corruption Scorecard:
   - Overall corruption score (e.g., numeric + label: “Clean”, “Questionable”, “Highly Corrupt”).
   - Top 10 sponsors list:
     - Sponsor name, industry, contribution amount/relative bar.
   - Average campaign donation.
   - Voting record:
     - List of recent votes with bill name, identifier, date, category, and position (Yea/Nay/etc.)
     - Optionally a short summary of the bill.

   Public Voting on Corruption:
   - Section where users can review recent news related to this politician.
   - For each news article:
     - Show title, source, date, short summary.
     - Link out to the full article.
     - Provide “Upvote Corrupt” and “Upvote Clean” buttons (or similar binary choice using corruption_votes table).
   - Display aggregated tallies:
     - Total “Corrupt” vs “Clean” votes.
     - Per-article breakdown.
   - Prevent duplicate votes per user/article using the UNIQUE constraint in corruption_votes.

   Recent News Articles:
   - Highlight the top 3 recent news articles (news_articles where is_recent = true) in a dedicated section.
   - Include "See more" link to show additional articles if available.

Make sure all UI components are responsive and fit into the existing ACP design system.

4. PROFILE CLAIM FLOW
--------------------------------
Implement the “Claim this profile” system so only the real representative can claim their profile:

1) From the Politician Profile page:
   - User clicks “Claim this profile”.
   - If they are not logged in, prompt login/signup first.
   - Then present a form to input an official email address.

2) Backend:
   - Create a politician_claims row with:
     - politician_id, user_id, official_email, randomly generated verification_token, status = "pending".
   - Send a verification email to that official email address with a secure link:
     - e.g., https://anticorruptionparty.us/verify-claim?token=XYZ

3) Verification endpoint:
   - When the link is clicked:
     - Look up the claim by verification_token.
     - If valid and not expired, set status = "approved" and verified_at = now.
     - Mark this user as the “claimed_owner” of the politician profile, e.g., add:
       - politicians.claimed_by_user_id (FK -> users.id, nullable)
   - Later, we can give claimed representatives limited edit powers (bio, photo, statements), but for now just store that the profile is verified/claimed.

4) UI:
   - If politician profile is claimed, display “Verified Representative” badge.
   - If user is the claimed representative, show a subtle note like “You are the verified owner of this profile” and hide the claim button.
